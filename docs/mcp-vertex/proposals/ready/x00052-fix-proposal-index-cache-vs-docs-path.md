# x00052 — `proposals/index.json` lives under docs/ but is regenerable state

## goal

Move the `proposals/index.json` file from `docs/mcp-vertex/proposals/`
into `.cache/mcp-vertex/proposals/` so the proposals plugin follows the
same "regenerable state lives under `cacheDir`, human-edited state
lives under `docsDir`" split that every other artefact already obeys
(locks, queue, registry, worktrees, counters, round-context digest).

Today `docs/mcp-vertex/proposals/index.json` is a 62 KB JSON file
**tracked in git** that is regenerated by `proposals_sync_proposals`
every time a proposal moves. Committing it is a 4-way anti-pattern:

1. It is a cache (regenerated on demand), not a source file.
2. It sits in `docs/`, the human-edited proposals root, where it
   visually crowds out the actual proposals.
3. The repo's own invariant — "the cache is ALWAYS the root cache,
   never per-folder" — already says regenerable state goes in
   `.cache/`. The `proposalIndexFile` is the one exception that
   escaped that rule.
4. Every commit that touches a proposal (rename, status flip, new
   proposal) forces a noisy diff in `index.json` — there are 136
   proposals in the registry, the JSON diff dwarfs the actual
   proposal diff every time.

## why

The comment at the top of
`plugins/proposals/src/lib/contracts/constants/default-path-layout.constant.ts:5-7`
states the rule the proposal enforces:

> "Cache artefacts (locks, queue, checkpoints, worktrees) live under
> `cacheDir`; human-edited proposals live under `docsDir`."

The actual layout then does the right thing for every artefact except
`proposalIndexFile`:

```ts
proposalIndexFile: joinRel(docsDir, 'proposals/index.json'),   // <-- exception
proposalIdCountersFile: joinRel(cacheDir, 'proposal-id-counters.json'),
worktreesDir: joinRel(cacheDir, '.worktrees'),
```

`proposalIdCountersFile` is right next to it in the same layout object
and **does** live under `cacheDir` — but the index, which is bigger,
regenerated more often, and read-only, is in `docs/`. The split is
inconsistent and the only thing keeping the inconsistency invisible
is that `proposals_sync_proposals` writes the file from the layout
function in both directions, so a host that follows the docs path
keeps working.

## non-goals

- **No new CLI flag** to choose where the index lives. The host gate
  is `--cacheDir`; `docsDir` is for proposals, `cacheDir` is for
  artefacts. The default layout is the contract.
- **No migration of user data.** The `index.json` is *derived state*
  — it can be regenerated from the proposal files in one
  `proposals_sync_proposals` call. The migration commit deletes the
  tracked copy; the on-disk copy regenerates on next host start.
- **No change to the proposal document format.** All proposal files
  (`fNNNN-*.md`, `xNNNN-*.md`, etc.) stay in `docs/`; only the
  derived index moves to `.cache/`.

## slices

### S1 — flip `proposalIndexFile` to live under `cacheDir`

File: `plugins/proposals/src/lib/contracts/constants/default-path-layout.constant.ts`.

- Change `proposalIndexFile: joinRel(docsDir, 'proposals/index.json')`
  to `proposalIndexFile: joinRel(cacheDir, 'proposals/index.json')`.
- Update the file's top comment to remove the "human-edited
  proposals live under `docsDir`" exception, or extend it to call
  out the index as a cache artefact explicitly. The latter is
  cleaner — it makes the rule more discoverable.

### S2 — ignore the docs/ location in `.gitignore`

File: `.gitignore`.

- Add the rule (next to the existing `apps/web/src/data/...`
  generated-file block, since that block already documents the
  pattern "generated, but the path is *not* under `.cache/` because
  the generator chose otherwise"):

  ```gitignore
  # proposals/index.json — regenerable cache artefact. Lives under
  # .cache/mcp-vertex/proposals/index.json since x00052; the legacy
  # docs/ path is ignored here so an old checkout cannot accidentally
  # commit it back.
  docs/mcp-vertex/proposals/index.json
  ```

- Update the `apps/web/src/data/manifests/` and `apps/web/src/data/capabilities.json`
  comments in the same block to cross-reference the proposals index
  rule, so the next person adding a generated file sees the pattern
  in one place.

### S3 — `git rm --cached` the tracked file

Single migration commit on `develop`:

```bash
git rm --cached docs/mcp-vertex/proposals/index.json
git commit -m "chore(proposals): untrack legacy docs/proposals/index.json (x00052)

The file is a regenerable cache artefact produced by
proposals_sync_proposals. It now lives under
.cache/mcp-vertex/proposals/index.json (S1 of x00052) and
should not be committed. The on-disk copy in docs/ is left
untouched so a running host does not crash; it will not be
re-created by sync_proposals (which now writes to the cache
path) but can be deleted manually."
```

- Verify the post-commit `git ls-files` for
  `docs/mcp-vertex/proposals/index.json` returns empty.
- Verify the on-disk copy is still there (we do not delete it
  in the commit; that's the user's call). A subsequent
  `bun run validate` regenerates the new cache path from the
  proposal files and leaves the docs copy stale but harmless.

### S4 — update tests that hardcode the docs path

The 12 test files found by `grep -rln "proposals/index.json"` fall
into two camps:

- **Tests that build a full `IHostPathLayout` via the constant.**
  These keep working unchanged as soon as S1 flips the layout —
  they read through `buildSwarmPaths(...)` and get the new path
  for free. No edit needed.
- **Tests that hardcode `docs/mcp-vertex/proposals/index.json` or
  assert the path ends with that suffix.** These need one of:
  - Replace the literal with `buildSwarmPaths(...).proposalIndexFile`,
    so the test follows the layout function.
  - Or, if the test deliberately exercises the legacy path (none
    does — checked), add a comment that the test is now stale and
    delete it.

I will not run the grep + edit loop blindly; the test pass after S1
+ the typecheck will tell me which ones broke, and I will fix those
specifically.

## acceptance criteria

- `bun run validate` is green (typecheck + lint + tests + verify:tools).
- `git ls-files docs/mcp-vertex/proposals/index.json` returns empty.
- `git ls-files .cache/mcp-vertex/proposals/index.json` is **not**
  the assertion (it must not be tracked either — the cache is
  always gitignored). What I do assert is: after a clean
  `bun tools/scripts/proposals/.../sync-proposals.script.ts` (or
  the runtime tool) on a fresh checkout, the new file appears at
  `.cache/mcp-vertex/proposals/index.json` and not at
  `docs/mcp-vertex/proposals/index.json`.
- Manual smoke: `rm docs/mcp-vertex/proposals/index.json &&
  bun tools/scripts/proposals/.../sync-proposals.script.ts &&
  test -f .cache/mcp-vertex/proposals/index.json &&
  test ! -f docs/mcp-vertex/proposals/index.json` exits 0.

## risks

- **External consumers that read `docs/mcp-vertex/proposals/index.json`**
  directly. There is no public API surface that lists them, but
  the file's tracked history means the path was public by
  convention. I will grep for direct readers in the next session
  if any user reports breakage.
- **Stale on-disk copy in docs/**. A host that already has
  `docs/mcp-vertex/proposals/index.json` on disk will keep using
  it until somebody deletes it (the runtime now writes to
  `.cache/`, so the docs copy is read-only-stale). The migration
  commit documents this; users can `rm` it manually. No silent
  corruption — `proposals_sync_proposals` will not re-create the
  docs copy because the layout no longer points there.
- **`proposalIdCountersFile` is in `cacheDir` already but its
  persistence story is the same as the index** (regenerated,
  in-memory authoritative). It is already in the right place —
  no change needed.
