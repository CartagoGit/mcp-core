## Resumen ejecutivo

`mcp-vertex` tiene una idea arquitectonica buena y bastante mejor que la media:
un nucleo pequeno, contratos tipados, plugins cargables, herramientas de
orientacion baratas y conocimiento bajo demanda. El proyecto compila, sus seis
paquetes se pueden empaquetar, el doctor carga los cinco plugins y la suite
actual termina con **267 tests pasados y 10 omitidos**.

Sin embargo, todavia no cumple su promesa mas importante: ser un runtime
agnostico, aislado por workspace y resistente a concurrencia. El plugin
`proposals`, precisamente el componente que coordina agentes, ignora las rutas
resueltas por el core, mezcla varios modelos de persistencia y permite carreras
de lectura-modificacion-escritura. Hay ademas falsos positivos del doctor,
comandos ejecutables con semantica incompleta y scaffolds que prometen agentes
que el host generado no puede soportar por si solo.

Mi valoracion global es **6,8/10** en el estado actual:

## FATAL - Errores criticos o de diseno

### 1. `proposals` ignora `cacheDir` y `docsDir`

El core construye correctamente `ctx.pluginCacheDir` y `ctx.pluginDocsDir`, pero
`plugins/proposals/src/index.ts` usa siempre `DEFAULT_PATH_LAYOUT`, construido
con `.cache` y `docs`. Por tanto:

- `--cacheDir=.state/mcp` no mueve locks, cola ni registros.
- `--docsDir=project-docs` no mueve las propuestas.
- dos instancias con configuraciones distintas pueden compartir estado sin
  pretenderlo;
- `overview` y `doctor` informan unas rutas mientras `proposals` escribe en
  otras.

Esto contradice README, contrato de plugins y comentarios del propio plugin.
Debe usar `buildSwarmPaths(ctx.pluginCacheDir, ctx.pluginDocsDir)` o definir de
forma explicita si `docsDir` ya representa la raiz final. Mientras no se
resuelva, no consideraria seguro ejecutar varias instancias sobre un mismo
workspace.

### 2. El lock de escritura no es un mutex real

`runAgentLockEngine` hace `read -> comprobar -> modificar -> write` sin exclusion
mutua y escribe directamente el JSON final. Dos claims simultaneos pueden leer
el mismo estado vacio, considerar ambos que no hay conflicto y sobrescribirse.
El resultado es exactamente lo que el lock pretende impedir: dos agentes creen
tener la misma propiedad de escritura.

Tambien pueden perderse heartbeats o releases concurrentes y un proceso
interrumpido durante `writeFile` puede dejar JSON parcial. Hace falta al menos:

1. lock interproceso por fichero o creacion exclusiva;
2. escritura temporal en el mismo directorio y `rename` atomico;
3. version/revision para detectar actualizaciones perdidas;
4. test de claims concurrentes reales desde procesos separados.

### 3. La cola persistente tampoco serializa sus escrituras

`enqueue`, `dequeue`, `appendToClosedTasks`, registro de subagentes y otras
operaciones repiten el patron de lectura-modificacion-escritura sin mutex
interproceso. `promoteOnRelease` tiene un mutex, pero solo en memoria y solo para
esa funcion. No protege frente a otra instancia MCP ni frente a `enqueue`.

Ademas, varias escrituras supuestamente atomicas crean el temporal en
`os.tmpdir()` y despues hacen `rename` al workspace. Ese rename puede fallar con
`EXDEV` si `/tmp` y el workspace estan en sistemas de archivos distintos. El
temporal debe estar junto al destino.

### 4. `task_queue report` lee el lock equivocado

La tool recibe rutas absolutas para cola y tareas cerradas, pero el action
`report` llama a `loadLockSnapshot(DEFAULT_PATH_LAYOUT.lockFile, ...)`. Esa ruta
es relativa al `cwd`, no al workspace ni a la configuracion inyectada. Puede
devolver backpressure verde cuando existen locks activos, o leer el lock de
otro proyecto. `ITaskQueuePaths` debe incluir `lockPath` y usarlo siempre.

## MUY MAL - Problemas serios

### 5. El doctor puede declarar sano un servidor que no arranca

Se comprobo con `--plugins=memory,memory`: `--check` devuelve `ok: true`, carga
dos veces el plugin y reporta 15 tools. Al ensamblar el servidor real falla con:

```text
Error: [mcp-vertex] duplicate registration id "memory_save"
```

El doctor debe ejecutar `planRegistrationOrder` y validar nombres reales de
tools, prompts, resources, knowledge y namespaces. El loader tambien deberia
deduplicar por plugin name/resolved specifier o rechazar duplicados con un error
claro.

### 6. No hay timeout para cargar ni registrar plugins

Un `import()` o `plugin.register()` que no resuelva bloquea indefinidamente el
arranque. La afirmacion "un plugin fallido nunca bloquea el boot" solo cubre
excepciones, no promesas colgadas. Debe existir timeout configurable,
diagnostico del plugin culpable y, si es posible, carga aislada.

### 7. Los acceptance commands no respetan bien el workspace ni el shell

`runAcceptanceCriteria` ejecuta sin `cwd` inyectado. Si el CLI se arranca desde
un directorio y `--workspace` apunta a otro, las validaciones se ejecutan en el
proyecto incorrecto. Ademas divide el comando con `split(/\s+/)`: rompe comillas,
espacios en rutas, pipes, redirecciones y variables. El comentario menciona
`bun -c`, pero el codigo no lo usa.

El timeout mata el proceso directo, no garantiza terminar descendientes. Para
evitar procesos zombis necesita grupo de procesos, senal escalonada y espera de
terminacion confirmada.

### 8. Scaffolding de agentes incoherente con el host generado

El host scaffold genera adapters que ordenan llamar primero a
`<prefix>_check_project_state`, `<prefix>_agent_lock` y
`<prefix>_continue_proposal`, pero el host generado registra inicialmente solo
la tool de scaffold. Tampoco integra automaticamente el plugin `proposals` en
ese host programatico. El resultado inicial contiene instrucciones imposibles
de cumplir.

Ademas fija `MiniMax-M3 (customendpoint)` como modelo por defecto. Una libreria
agnostica no debe imponer un proveedor/modelo concreto; debe omitir el campo o
pedirlo como opcion.

### 9. Corrupcion silenciosa convertida en estado vacio

Memory, cola, registro de subagentes y tareas cerradas suelen tratar JSON
corrupto como `[]` o estado vacio. Esto mantiene el servidor vivo, pero puede
provocar perdida de datos, reasignaciones dobles y decisiones falsas. La
resiliencia correcta es conservar el archivo corrupto, emitir un error
estructurado y ofrecer recuperacion; no fingir que nunca existio.

## REGULAR - Funciona, pero necesita mejora

### 10. `rules` conserva un manifest obsoleto indefinidamente

Los presets se reescriben al arrancar, pero `rules-map.json` solo se crea si no
existe. Cambiar `mode`, overrides, paquetes o frameworks no actualiza el mapa.
Luego las tools confian en el JSON sin validarlo. Conviene separar manifest
generado de overrides humanos y regenerar por hash/version.

### 11. `rules` no comprueba realmente todas las reglas anunciadas

`check_rules` devuelve un comando ESLint, aunque el mapa tambien anuncia
typecheck configs. No ejecuta ni propone el comando TypeScript correspondiente.
Usa `eslint` desnudo, que depende del PATH, y concatena rutas sin escaping. Es
una guia util, no todavia un verificador fiable.

### 12. `quality` bloquea el event loop y pierde semantica de error

Usa `execSync`, por lo que un test largo congela todas las peticiones MCP hasta
10 minutos. El timeout existe, pero no se distingue de un exit 1 y el limite de
buffer por defecto puede hacer fallar comandos verbosos. Deberia usar spawn
asincrono, salida acotada en streaming, codigo de timeout explicito y politica
de concurrencia/cancelacion.

### 13. `git` oculta todos los errores

Un directorio que no es repo, ausencia de git o un timeout terminan como salida
vacia, indistinguible de un repo limpio. Tampoco hay timeout y `git_log.limit`
acepta numeros negativos, decimales o enormes. Debe devolver `{ok:false,
reason}` y limitar `1..100`.

### 14. `memory` no tiene limites ni escritura atomica

No limita longitud de titulo/cuerpo, numero de tags, numero total de notas ni
tamano del fichero. `limit` puede ser negativo. Dos saves concurrentes pueden
perder una nota y un corte durante escritura puede vaciar toda la memoria.

### 15. El contrato de IDs impide composicion generica

`planRegistrationOrder` exige IDs globalmente unicos, aunque dos plugins con
namespaces distintos podrian tener legitimamente una tool interna `status`.
Para plugins arbitrarios, la identidad deberia incluir plugin/namespace, o el
loader deberia reescribir IDs internos. Ahora la composicion funciona porque
los cinco plugins incluidos coordinan manualmente sus nombres.

### 16. La API `/lib/*` es demasiado abierta

Los README llaman estable a `/public` y menos estable a `/lib/*`, pero todos los
internals se publican mediante wildcard exports. Consumidores como `affairs`
ya dependen masivamente de ellos. Esto dificulta evolucionar esquemas y
persistencia. Hace falta versionar contratos publicos reales y reducir deep
imports antes de `1.0`.

### 17. Cobertura desconocida y tests demasiado unitarios

La suite es rapida y amplia en funciones puras, pero no hay metrica de coverage
ni CI visible en este workspace. Faltan pruebas de:

- dos procesos reclamando el mismo fichero;
- dos instancias escribiendo cola/memory/registry;
- rutas custom de core aplicadas a todos los plugins;
- doctor versus arranque real;
- proceso hijo con descendientes al expirar timeout;
- JSON truncado y recuperacion;
- instalacion en un proyecto consumidor limpio.

## COMO DEBE ESTAR - Correcto y funcional

### 18. Core pequeno y dependencias contenidas

El core depende solo del SDK MCP y Zod. No importa codigo de dominio y mantiene
la composicion en interfaces pequenas. Esta es la direccion correcta.

### 19. Workspace provider explicito

`createWorkspacePathProvider` normaliza una raiz absoluta y memoiza
resoluciones. El contrato es sencillo, inyectable y facil de probar. El fallo
esta en plugins que lo eluden, no en esta pieza.

### 20. Registro determinista y fail-fast

`planRegistrationOrder` conserva orden, soporta anchors y rechaza duplicados o
anchors desconocidos. Para una unica secuencia controlada es claro y fiable.

### 21. Configuracion estructurada y validable

La combinacion CLI/config/plugin options esta bien separada. Zod en
`optionsSchema` permite fallar antes de registrar. El modo doctor es una buena
interfaz, aunque debe ampliar sus comprobaciones.

### 22. Empaquetado por paquetes

Los seis paquetes producen tarballs pequenos y con `files` limitado a source,
README y licencia. El core queda en unos 111 KB desempaquetado y los plugins
pequenos entre 10 y 40 KB; `proposals` ronda 310 KB. Es razonable para paquetes
TypeScript ejecutados con Bun.

## BIEN - Por encima de lo esperado

### 23. `overview` como entrada de cold start

Una unica llamada devuelve identidad, plugins, tools, knowledge y siguiente
accion. Reduce exploracion ciega y llamadas repetidas. Es una optimizacion real
de tokens.

### 24. Knowledge lazy y resources MCP

Listar solo IDs/titulos y recuperar el cuerpo bajo demanda es mejor que inyectar
toda la documentacion en cada turno. Exponer lo mismo como resources permite a
clientes compatibles cachearlo.

### 25. Respuestas compactas en el core

`toolJson`, `toolOk` y `toolError` usan JSON compacto y envelope accionable.
Esto facilita modelos pequenos y reduce interpretacion ambigua. Falta migrar
las tools antiguas de `proposals`, que aun usan pretty JSON con tabs.

### 26. Plugins pequenos con responsabilidad clara

`git`, `memory`, `quality` y `rules` son comprensibles y reutilizables. Sus
builders reciben dependencias inyectadas, lo que facilita tests y hosts
programaticos.

### 27. Buen uso de funciones puras en `proposals`

Presupuestos, politicas de continuidad, evaluacion de paralelismo, parsing y
closure report separan bien calculo de I/O. Es la parte mas madura del plugin.

## MUY BIEN - Ejecucion excelente

### 28. Politicas explicitas contra reintentos ciegos

Los payloads de lock indican `Do not retry`, proponen otra slice y clasifican
el conflicto. `continuity-policy`, recovery y backpressure expresan estados con
vocabulario semantico. Conceptualmente es una defensa muy buena contra loops de
agentes.

### 29. Round context con fingerprinting

El digest, hashes y deteccion de staleness evitan releer todo el workspace al
reanudar una ronda. La escritura temporal junto al destino si esta bien
planteada. Es una optimizacion util de continuidad y tokens.

### 30. Parsing y errores de propuestas

La validacion por capas y los codigos semanticos permiten respuestas mas
accionables que excepciones genericas. La suite cubre muchos casos limite de
presupuesto, frontmatter, paralelismo y continuidad.

## PERFECTO - Piezas de referencia

No calificaria el sistema completo como perfecto. Si hay que destacar piezas
que, aisladas, estan practicamente en ese nivel:

### 31. Contrato minimo `IMcpPlugin`

`name`, metadata opcional, schema de opciones y `register(ctx)` bastan para
componer tools, prompts, resources, knowledge y skills. Es facil de entender y
no obliga a herencia ni contenedores complejos.

### 32. Separacion entre API publica y builders inyectables

Los plugins exponen builders reutilizables y tipos, no solo un default export
magico. Esto permite usar el sistema por CLI o como libreria y facilita la
adopcion incremental.

### 33. Velocidad de la suite actual

Los 277 casos se procesan en menos de cuatro segundos en esta maquina. Para un
proyecto con bastante logica de estado, esa velocidad favorece ejecutar tests
con frecuencia y reduce el coste operativo de agentes.

## Analisis de eficiencia de tokens para agentes

### Lo que reduce tokens

- `overview` evita descubrir tools una por una.
- knowledge lazy evita cargar manuales completos.
- `auto_work` concentra siguiente propuesta y plan en una llamada.
- git diff stat y listas de cambios son mas baratas que diffs completos.
- caps de closed tasks y tails de quality limitan crecimiento.
- round-context y digests reducen reconstruccion tras reanudar.

### Lo que desperdicia tokens

- `proposals` usa bastante JSON pretty-print; conviene envelope compacto.
- `overview` enumera siempre todas las tools. Con 30 tools sigue siendo
  aceptable, pero deberia ofrecer `compact:true` o agrupacion por tags cuando
  crezca.
- reglas materializa convenciones de todos los presets presentes, incluso si
  el agente trabaja en una sola area.
- el blueprint genera una tool por cada quality role aunque el plugin quality
  ya resuelve ese problema: hay duplicacion conceptual y de contexto.
- instrucciones generadas repiten el mismo workflow en prompt, skill, agent y
  copilot instructions. Una referencia corta a knowledge versionado seria mas
  barata y evitaria drift.

### Veredicto de tokens

Para sesiones de orientacion y trabajo simple, probablemente ahorra tokens.
Para swarm complejo, el ahorro solo sera real cuando el estado persistente sea
fiable: un lock perdido o un manifest stale obliga a reauditar y consume mucho
mas de lo ahorrado. Estimacion cualitativa actual: **15-30 % de ahorro** en cold
start bien configurado; beneficio incierto bajo concurrencia.

## Analisis de posibles bucles y bloqueos

### Bucles mitigados correctamente

- conflicto de lock devuelve una accion alternativa en vez de recomendar
  polling inmediato;
- continuidad y staleness tienen politicas explicitas;
- budgets limitan iteraciones, tool calls y premium calls;
- backpressure clasifica green/amber/red;
- el flujo promueve slices pequenas y validacion antes de cierre.

### Bucles o bloqueos posibles

1. **Plugin colgado en import/register:** boot infinito, sin timeout.
2. **Quality sincrono:** servidor completo no responde mientras corre el gate.
3. **Acceptance descendants:** un timeout puede dejar procesos hijos vivos.
4. **Lost update del lock:** dos agentes trabajan a la vez creyendo poseer el
   mismo fichero.
5. **Waiters sin promocion fiable:** la promocion depende de estado y formas de
   lock no transaccionales; puede quedar una tarea esperando para siempre.
6. **Estado corrupto tratado como vacio:** puede recrear trabajo ya realizado y
   causar ciclos de reasignacion.
7. **Doctor falso positivo:** automatizaciones pueden reintentar un servidor que
   nunca arrancara.
8. **Heartbeat manual:** no existe un watchdog autonomo que renueve o cierre
   ownership; depende de que el agente recuerde llamar a la tool.

No puede afirmarse que "no hay bucles ni bloqueos posibles". El proyecto tiene
buenas politicas anti-loop, pero le falta una capa runtime transaccional y
timeouts sistematicos.

## Skills, herramientas y agentes faltantes

### Skills recomendadas

1. **`mcp-vertex-plugin-authoring`**: contrato, namespaces, schemas, envelopes,
   timeouts, rutas y checklist de publicacion.
2. **`mcp-vertex-state-recovery`**: como diagnosticar y recuperar lock, queue,
   registry, memory y digests corruptos sin perder evidencia.
3. **`mcp-vertex-concurrency`**: invariantes de ownership, atomics, CAS, pruebas
   multiproceso y reglas para no mezclar stores.
4. **`mcp-vertex-token-budgeting`**: cuando usar overview, resources, summaries,
   pagination y payload compacto.
5. **`mcp-vertex-security`**: modelo de confianza para plugins y comandos,
   allowlists, redaccion de secretos y limites de filesystem/proceso.

Estas skills deberian vivir como knowledge/resources versionados y ser
materializables para clientes que soporten skills nativas. No conviene cargar
las cinco siempre.

### Herramientas recomendadas

1. **`doctor_runtime`**: ensambla realmente el servidor sin conectar stdio,
   detecta IDs/nombres/URIs duplicados y valida handlers.
2. **`state_health`**: valida schemas, ownership, revisiones, archivos temp y
   coherencia lock/queue/registry; nunca corrige por defecto.
3. **`state_repair`**: reparacion explicita con dry-run, backup y journal.
4. **`plugin_capabilities`**: muestra permisos efectivos: filesystem,
   subprocess, network, mutaciones y timeouts.
5. **`compact_status`**: estado unificado de git, quality, proposals y locks en
   un payload pequeno, con campos seleccionables.
6. **`cancel_operation`**: cancelacion de quality/acceptance y operaciones
   largas por operation id.
7. **`export_diagnostics`**: bundle redactado para incidencias, sin secretos ni
   cuerpos completos de memoria.

### Agentes recomendados

No agregaria mas agentes al flujo normal. Cinco roles para cualquier proyecto
es demasiado por defecto. Recomiendo:

- **modo simple:** un agente generalista, sin swarm;
- **modo coordinado:** orchestrator + implementation runner;
- **modo critico:** anadir delivery verifier;
- proposal guardian y technical investigator solo bajo demanda.

Si se incorpora uno nuevo, que sea un **runtime/state custodian** no editor:
diagnostica invariantes, corrupcion y procesos colgados. No debe reparar sin
autorizacion ni competir por archivos de producto.

## Analisis individual de plugins

### `proposals` - 5,2/10

Es el plugin con mas valor diferencial y tambien el mayor riesgo. Las funciones
puras son buenas; la persistencia y el wiring de rutas no estan al mismo nivel.
No lo habilitaria para escritura multiagente real hasta corregir los cuatro
hallazgos fatales.

### `rules` - 6,8/10

Buena idea como orientador, pero el nombre promete enforcement que en realidad
es asesoria. Necesita invalidacion del manifest, validacion runtime y comandos
de lint/typecheck portables.

### `quality` - 7,0/10

Resuelve una necesidad real con una interfaz pequena. Debe migrar a ejecucion
asincrona, cancelable y con limites de salida. Tambien necesita un modelo de
confianza explicito porque ejecuta comandos del repositorio.

### `git` - 7,5/10

Pequeno, barato en tokens y read-only. Su mayor defecto es confundir error con
salida vacia. Con timeout, limites y errores estructurados seria excelente.

### `memory` - 6,9/10

Util para continuidad y simple de entender. Falta robustez de store, quotas,
redaccion de secretos y concurrencia. No debe presentarse como durable mientras
una escritura parcial pueda borrar todo el historial logico.

## Analisis del plugin `proposals` - complejidad frente a necesidad

La complejidad esta justificada solo para proyectos que de verdad usan varios
agentes concurrentes y propuestas ejecutables. Para trabajo serial, locks,
registry, queue, round context, zombie recovery, naming y closure gates pueden
costar mas contexto y mantenimiento del que ahorran.

Recomiendo dividirlo internamente en capacidades activables:

- `proposals-store`: documentos, index, sync y acceptance;
- `proposals-locks`: ownership transaccional;
- `proposals-queue`: dependencias y backpressure;
- `proposals-swarm`: nombres, round context y continuidad.

El preset `proposals` podria cargar las cuatro, pero un proyecto simple podria
usar solo store. Esto reduce tools visibles, superficie de fallo y tokens.

## Recomendaciones prioritarias

### P0 - Antes de usar swarm en serio

1. Hacer que `proposals` derive todas las rutas desde `ctx`.
2. Implementar store transaccional comun con temp local, revision y mutex
   interproceso para lock/queue/registry/closed tasks/memory.
3. Inyectar `lockPath` en task queue report.
4. Anadir tests multiproceso de claims y escrituras simultaneas.

### P1 - Fiabilidad operativa

5. Hacer que doctor ensamble y valide el servidor real.
6. Deduplicar plugins y validar namespaces/nombres.
7. Timeouts para import/register; subprocess async, cancelable y con process
   groups.
8. Ejecutar acceptance con `cwd=workspace`, parser de argv explicito o shell
   declarada.
9. Diferenciar missing, corrupt y empty; backups antes de repair.

### P2 - Calidad de producto

10. Corregir scaffolds para que las tools prometidas existan y eliminar el
    modelo hardcoded.
11. Regenerar `rules-map` por fingerprint y validar su schema.
12. Unificar envelopes compactos y paginar/listar por tags.
13. Cerrar gradualmente `/lib/*` y definir compatibilidad semver.
14. CI con typecheck, tests, pack/install smoke y cobertura minima por paquete.

### P3 - Evolucion

15. Separar `proposals` por capacidades.
16. Publicar skills de authoring, recovery, concurrency y security.
17. Introducir capability metadata para que el usuario sepa que plugins
    escriben, ejecutan procesos o leen datos sensibles.

## Valoracion global

La base conceptual es buena y merece continuar. El core, la carga de plugins,
la orientacion low-token y las funciones puras muestran criterio de diseno. No
es un proyecto "fatal" ni sobredisenado por definicion; es un proyecto joven
que ha avanzado mas deprisa en politicas y features que en garantias de
persistencia.

La prioridad no deberia ser sumar mas herramientas o mas agentes. Debe ser
hacer verdaderas tres promesas ya escritas en la documentacion: **rutas
inyectadas**, **operaciones idempotentes** y **coordinacion sin carreras**. Una
vez cerradas, la arquitectura puede convertirse en una plataforma MCP muy
solida y realmente reutilizable.

**Nota final: 6,8/10.**  
**Potencial tras P0/P1: 8,5/10.**

### Que faltaria para alcanzar un 10/10

P0 y P1 convertirian el runtime en una base fiable, pero un 10/10 exige algo
mas que corregir defectos: debe demostrar madurez de producto, seguridad,
portabilidad y operacion sostenida. Para llegar ahi faltaria:

1. **Garantias transaccionales demostrables.** Un unico subsistema de estado
   con locks interproceso, revision/CAS, journal o WAL, recuperacion tras crash,
   backups y migraciones versionadas. Sus invariantes deben estar probadas con
   procesos concurrentes, reinicios y fault injection.
2. **Aislamiento y seguridad de plugins.** Declaracion de capabilities por
   plugin, permisos minimos, allowlists para comandos y rutas, redaccion de
   secretos, limites de recursos y una opcion real de ejecutar plugins no
   confiables fuera del proceso principal.
3. **Operaciones largas totalmente controlables.** Ejecucion asincrona,
   progress events, operation IDs, cancelacion, timeouts jerarquicos, cierre de
   todo el arbol de procesos y proteccion contra saturacion del servidor.
4. **Contratos publicos estables.** Eliminar la dependencia habitual de
   `/lib/*`, definir semver y politica de deprecacion, tests de compatibilidad y
   migraciones automaticas de formatos persistidos entre versiones.
5. **Doctor con paridad total respecto al runtime.** Debe verificar el mismo
   ensamblado que arrancara, schemas, nombres, URIs, permisos, ejecutables,
   rutas escribibles, versiones compatibles y una llamada MCP smoke real.
6. **Observabilidad de produccion.** Logs estructurados y redactados, metricas
   de latencia/errores/tokens, trazas por tool y operacion, diagnosticos
   exportables y health checks que distingan degradacion de fallo.
7. **Eficiencia de contexto medible.** Benchmarks reproducibles contra un
   agente sin MCP: tokens de cold start, llamadas por tarea, tiempo hasta la
   primera edicion correcta y tasa de reintentos. Los payloads deben soportar
   compactacion, filtros, paginacion y presupuestos de salida.
8. **Composicion adaptativa.** Presets `minimal`, `standard` y `swarm`, carga de
   capacidades bajo demanda y descubrimiento progresivo para no exponer 30
   tools cuando una tarea solo necesita tres.
9. **Scaffolding realmente listo para usar.** Lo generado debe compilar,
   arrancar, incluir solo tools existentes, detectar el cliente/modelo sin
   hardcodes y superar automaticamente tests e instalacion en un repositorio
   limpio.
10. **Calidad multiplataforma y multi-runtime.** CI en Linux/macOS/Windows,
    workspaces con espacios y symlinks, filesystems distintos, Bun soportado de
    forma explicita y una decision documentada sobre compatibilidad con Node.
11. **Pruebas de sistema y caos.** Matriz end-to-end con varios clientes MCP,
    multiples servidores simultaneos, plugins defectuosos, JSON truncado,
    disco lleno, permisos denegados, procesos colgados y recuperacion sin
    perdida silenciosa.
12. **Documentacion operativa verificable.** Tutorial minimo, arquitectura,
    threat model, runbooks de recovery, guia de autor de plugins y ejemplos que
    se ejecuten en CI para impedir drift.

Con P0/P1 el proyecto podria ser **muy bueno y fiable**. Para un 10/10 deberia
ademas demostrar, mediante benchmarks y pruebas de sistema, que sigue siendo
seguro, portable, observable y eficiente cuando hay concurrencia real, fallos
parciales, plugins de terceros y upgrades. Ese seria el salto de una biblioteca
prometedora a una plataforma MCP de referencia.
