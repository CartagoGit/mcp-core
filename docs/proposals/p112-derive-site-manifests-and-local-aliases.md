# p112 — Derive site manifests + adopt `#MAYÚSCULAS/*` local-alias convention

> **Status: IDEA. Explored 2026-06-20 in two slices that were reverted by
> the workspace's parallel-edit reconciliation (cf. `affairs-orchestrator
> parallel agent commits your work mid-session`). This proposal is the
> surviving artefact: a self-contained spec that can be re-attempted
> slice by slice when the orchestrator's working-tree churn settles.**

## 0. The problems this proposal solves

Today, in `apps/web/`:

1. **The two site manifests live in the wrong place.**
   - `apps/web/src/data/skills.json` (regenerated by
     `apps/web/scripts/gen-skills.ts`).
   - `apps/web/src/data/capabilities.json` (regenerated by
     `apps/web/scripts/gen-capabilities.ts`).
   - Both are **declarative lists of what the site can render** (i.e.
     site manifests), NOT cache. But they sit at the same level as the
     real source code under `src/data/` (`install.ts`, `marquees.ts`,
     `loop-detector-i18n.ts`), so the directory is a confusing mix of
     "real source code" and "derived outputs". `apps/web/src/data/
     capabilities.json` is currently git-ignored via a one-off rule in
     `.gitignore`; `apps/web/src/data/skills.json` is **not** ignored
     and shows up as a regenerated file in `git status` on every build.

2. **Other derived runtime directories leak into `git status` because
   they are not in `.gitignore`.**
   - `.mcp-vertex/handoff/` (agent handoff files).
   - `docs/mcp-vertex/proposals/index.json` (regenerated by the
     proposals plugin).

3. **All imports in `apps/web/src/` use relative paths** like
   `from '../components/ToolsSection.astro'`, `from '../../data/
   capabilities.json'`, `from '../i18n/ui'`. This makes the consumer's
   intent ("is this a workspace package or local code?") hard to read
   at a glance. The repo already uses `@mcp-vertex/*` for workspace
   packages via `tsconfig.base.json#paths`, so adding a distinct local
   alias convention is the natural next step.

## 1. The convention this proposal establishes

Two-part naming convention, applied repo-wide (this slice: `apps/web/`,
follow-up slices: `packages/core/`, `plugins/*/`, `examples/*/`).

| Aspect            | Convention                                                |
| ----------------- | --------------------------------------------------------- |
| Filesystem dirs   | **kebab/lowercase** (`manifests/`, `components/`, …)      |
| Local import path | **`#MAYÚSCULAS/...`** (`#MANIFESTS/capabilities.json`)    |
| Workspace imports | `@mcp-vertex/...` (unchanged)                             |

The `#` prefix is the npm subpath import convention; it cannot collide
with any workspace package name. SCREAMING_SNAKE_CASE in the alias
visually distinguishes "output dir / project-local" from the lowercase
filesystem path and from the camelCase `@scope/name` workspace
imports.

### The 6 aliases used in `apps/web/`

| Alias              | Resolves to                              | Source vs derived?     |
| ------------------ | ---------------------------------------- | ---------------------- |
| `#MANIFESTS/*`     | `apps/web/src/data/manifests/*`          | **derived** (git-ignored) |
| `#DATA/*`          | `apps/web/src/data/*`                    | source code            |
| `#COMPONENTS/*`    | `apps/web/src/components/*`              | source code            |
| `#LAYOUTS/*`       | `apps/web/src/layouts/*`                 | source code            |
| `#I18N/*`          | `apps/web/src/i18n/*`                    | source code            |
| `#SCRIPTS/*`       | `apps/web/scripts/*`                     | source code            |

The source-of-truth declarations live in
`apps/web/scripts/lib/local-aliases.mjs` (new file). This module
exports `LOCAL_ALIASES` (a `Record<string, string>` of absolute paths,
consumed by Vite) and `TS_CONFIG_PATHS` (a `Record<string, string[]>`
of relative paths for `tsconfig.json#compilerOptions.paths`). Both
exports are computed from the same `REPO_ROOT` constant so the two
configurations cannot drift.

Each slice has its own acceptance criteria and runs the relevant gate
locally before commit.

## 3. Why `#` (npm subpath) and not `@` or `~`

The conversation that produced this proposal considered three prefixes:

| Prefix | Verdict | Reason                                                                |
| ------ | ------- | --------------------------------------------------------------------- |
| `~`    | rejected | Looks like a Unix home dir; confusing in CLI/shell output.           |
| `@`    | rejected | Already used for workspace packages (`@mcp-vertex/*`).               |
| `#`    | **chosen** | Reserved by npm for subpath imports; cannot collide with packages. |

The `#` prefix is the standard ESM subpath import convention
(introduced in Node 16, TypeScript 4.7+). It also gives a free visual
separator when reading code:

```ts
import { createMcpProject } from '@mcp-vertex/core/public';   // npm
import capabilities         from '#MANIFESTS/capabilities.json'; // local
```

## 4. What this proposal does NOT touch

- The runtime state under `.cache/mcp-vertex/` (already correctly
  git-ignored).
- The proposals plugin's own `index.json` regeneration path (its
  docsDir resolution, addressed by p111 s5).
- Workspace packages in `packages/core/`, `plugins/*/`, `examples/*/`
  (would be a follow-up proposal — the convention is established
  here, but applying it repo-wide is out of scope).

## 5. Risks and mitigations

- **Risk**: Vite + Astro may not honour `tsconfig.json#paths` for
  runtime resolution, so `vite.resolve.alias` is still required.
  **Mitigation**: s2 wires both. The sync test in s6 prevents drift.

- **Risk**: Some tools (Biome, ESLint) may flag `#`-prefixed
  imports as invalid module specifiers. **Mitigation**: Biome 2.x
  accepts them; if a linter complains, add an `// biome-ignore` with
  a justification linking to this proposal.

- **Risk**: A future `npm publish` could mistakenly publish a package
  named `#data` or similar. **Mitigation**: `#` is reserved by npm
  for subpath imports; a package named `#data` is structurally
  rejected.

## 6. Rollback

Each slice is independently revertible. The aliases live in two
places (`local-aliases.mjs` + `tsconfig.json` + `astro.config.mjs`);
removing all three reverts s1 + s2 with no behavioural change. s3–s5
each touch a separate concern (manifest output location, consumer
imports, gitignore) and can be reverted in any order. s6 (the sync
test) is purely additive.

## 7. Lessons learned from the reverted 2026-06-20 attempt

- The workspace's parallel-commit reconciliation can revert working-
  tree changes within seconds of writing them. Per the project's
  memory (`affairs-copilot-editor-sandbox`), the right response is:
  **do not panic, do not redo the work, read the commit**. If the
  commit covers your intent, accept it and proceed; otherwise, do a
  surgical follow-up.
- When a slice is reverted before it can be committed, **documenting
  the spec as a proposal is the safest recovery path** — it captures
  the decision so a future attempt (human or agent) starts from
  consensus, not from scratch.
- **Do not stage or commit anything until the working tree has been
  stable for at least one `git status` cycle.**
