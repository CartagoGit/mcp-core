# proposals प्लगइन के साथ शुरुआत

यह ट्यूटोरियल एक खाली workspace से शुरू होता है और प्रस्ताव →
स्लाइस → कार्यान्वयन → बंद करने के पूरे चक्र के साथ समाप्त होता है,
जिसमें फ़ाइल म्यूटेक्स अनुशासन बरकरार रहता है। यह मानता है कि
`proposals` प्लगइन सक्षम है (JSON snippet के लिए
`plugins/proposals/README.md` देखें)।

## 0. मानसिक मॉडल

एक **प्रस्ताव** एक frontmatter हेडर वाली markdown फ़ाइल है। एक
**स्लाइस** उसके अंदर एक क्रमांकित खंड है। प्लगइन प्रति स्लाइस दो
लेखकों का समन्वय करता है: एक दावा करता है, एक जारी करता है।
`auto_work` उच्च-स्तरीय प्रवेश बिंदु है: "मुझे आगे क्या करना चाहिए?"।

```
docs/mcp-vertex/proposals/
├─ index.json          (sync_proposals द्वारा पुनर्जनित)
├─ p<N>-<शीर्षक>.md  (एक प्रस्ताव)
│  ├─ ## Slices
│  │  ├─ s1-claim
│  │  ├─ s2-implement
│  │  └─ s3-close
```

## 1. `auto_work` से शुरू करें

`auto_work` पूरे प्रस्ताव स्टोर में अगला कार्ययोग्य स्लाइस देता है,
एक कॉम्पैक्ट क्रमबद्ध योजना के साथ। योजना को शब्दशः निष्पादित
किया जाना चाहिए — कदम सुधारे बिना।

```json
// MCP टूल कॉल
{ "tool": "proposals_auto_work", "args": {} }

// सामान्य प्रतिक्रिया (संक्षिप्त)
{
  "state": "work",
  "proposalId": "l110",
  "sliceId": "s1-claim",
  "steps": [
    "docs/mcp-vertex/proposals/l110-…md खोलें और अगला परमाणु स्लाइस चुनें।",
    "इसकी फ़ाइलें दावा करें: proposals_agent_lock { action: \"claim\", … }।",
    "बिल्कुल उस स्लाइस को लागू करें — दावा की गई फ़ाइलों के बाहर कुछ नहीं।",
    "प्रोजेक्ट गेट के अनुसार मान्य करें (यदि मौजूद हो तो get_validation_matrix देखें)।",
    "प्रस्ताव में प्रगति चिह्नित करें, फिर proposals_sync_proposals।",
    "जारी करें: proposals_agent_lock { action: \"release\", task_id }।"
  ]
}
```

## 2. स्लाइस की फ़ाइलें दावा करना

`proposals_agent_lock` टूल रिकॉर्ड करता है कि स्लाइस की अवधि के
दौरान कौन से पथ किसके पास हैं। दावे के बिना, `sync_proposals` स्लाइस
को पूर्ण के रूप में चिह्नित करने से इनकार करेगा।

```json
{
  "tool": "proposals_agent_lock",
  "args": {
    "action": "claim",
    "files": [
      "apps/web/src/components/PluginPage.astro",
      "apps/web/src/data/capabilities.json"
    ]
  }
}
```

प्रतिक्रिया एक `task_id` लेकर आती है जिसे जारी होने तक रखना होगा।
एक ही फ़ाइल का दावा करने वाले दो एजेंट ⇒ संघर्ष, कोई प्रगति नहीं।
म्यूटेक्स fs-समर्थित है (सलाहकार नहीं) और प्रक्रिया पुनरारंभ के बाद
भी बना रहता है।

## 3. स्लाइस लागू करें, फिर मान्य करें

केवल दावा की गई फ़ाइलें संपादित करें। प्रोजेक्ट का गेट चलाएं:

```bash
bun run validate
```

यदि गेट विफल हो, तो स्लाइस ठीक करें — चुपचाप दावे का विस्तार न करें।

## 4. प्रगति चिह्नित करें और सिंक करें

`sync_proposals` प्रस्ताव फ़ाइलें पढ़ता है, उनके frontmatter + स्लाइस
योजना को मान्य करता है, और `index.json` पुनर्निर्माण करता है। यह
सस्ता और idempotent है।

```json
{ "tool": "proposals_sync_proposals", "args": {} }
```

## 5. स्लाइस बंद करें

```json
{
  "tool": "proposals_close_slice",
  "args": {
    "proposalId": "l110",
    "sliceId": "s1-claim"
  }
}
```

यह प्रस्ताव में स्लाइस स्थिति को `done` में पुनर्लिखता है, लॉक हटाता
है, और इंडेक्स पुनः सिंक करता है। फिर `auto_work` को फिर से कॉल
करें — यह अगला स्लाइस लौटाएगा (या `state: "idle"` यदि स्टोर
समाप्त हो गया हो)।

## सामान्य त्रुटियां

- **दावे से बाहर फ़ाइलें संपादित करना**: `sync_proposals` स्लाइस को
  पूर्ण चिह्नित करने से इनकार करेगा। अपने स्वयं के दावे के साथ दूसरा
  स्लाइस उपयोग करें, या प्रस्ताव विभाजित करें।
- **`sync_proposals` छोड़ना**: इंडेक्स पुराना हो जाता है। अगला एजेंट
  "अगला स्लाइस" मांगता है और गलत प्राप्त करता है।
- **जारी करना भूलना**: पुराना लॉक अगले ऑर्केस्ट्रेटर को `staleMs`
  तक (डिफ़ॉल्ट 30 सेकंड) ब्लॉक करता है। साफ़ करने के लिए
  `proposals_agent_lock { action: "gc" }` कॉल करें।
