# Getting started with the proposals plugin

This walkthrough takes a fresh workspace and ends with a working
proposal → slice → implementation → close loop, with the file
mutex discipline intact. Assumes the `proposals` plugin is enabled
(see `plugins/proposals/README.md` for the JSON snippet).

## 0. The mental model

A **proposal** is a markdown file with a frontmatter header. A
**slice** is a numbered section inside it. The plugin coordinates
two writers per slice: one claim, one release. `auto_work` is the
high-level "what should I do next" entry point.

```
docs/mcp-vertex/proposals/
├─ index.json          (regenerated by sync_proposals)
├─ p<N>-<title>.md     (a proposal)
│  ├─ ## Slices
│  │  ├─ s1-claim
│  │  ├─ s2-implement
│  │  └─ s3-close
```

## 1. Start with `auto_work`

`auto_work` returns the next actionable slice across the whole
proposal store, with a compact ordered plan. The plan is meant to
be executed verbatim — no improvising steps.

```json
// MCP tool call
{ "tool": "proposals_auto_work", "args": {} }

// Typical response (truncated)
{
  "state": "work",
  "proposalId": "l110",
  "sliceId": "s1-claim",
  "steps": [
    "Open docs/mcp-vertex/proposals/l110-…md and pick the next atomic slice.",
    "Claim its files: proposals_agent_lock { action: \"claim\", … }",
    "Implement exactly that slice — nothing outside the claimed files.",
    "Validate per the project gate (see get_validation_matrix if present).",
    "Mark progress in the proposal, then proposals_sync_proposals.",
    "Release: proposals_agent_lock { action: \"release\", task_id }."
  ]
}
```

## 2. Claim the slice's files

The `proposals_agent_lock` tool records who owns which paths for
the duration of a slice. Without a claim, `sync_proposals` will
refuse to mark the slice done.

```json
{
  "tool": "proposals_agent_lock",
  "args": {
    "action": "claim",
    "files": [
      "apps/web/src/components/PluginPage.astro",
      "apps/web/src/data/capabilities.json"
    ]
  }
}
```

The response carries a `task_id` you must keep until release. Two
agents claiming the same file ⇒ conflict, no progress. The mutex
is `fs`-backed (not advisory) and survives process restarts.

## 3. Implement the slice, then validate

Edit only the claimed files. Run the project's gate:

```bash
bun run validate
```

If the gate fails, fix the slice — do not silently widen the claim.

## 4. Mark progress and sync

`sync_proposals` reads the proposal files, validates their
frontmatter + slices plan, and rebuilds `index.json`. It's cheap
and idempotent.

```json
{ "tool": "proposals_sync_proposals", "args": {} }
```

## 5. Close the slice

```json
{
  "tool": "proposals_close_slice",
  "args": {
    "proposalId": "l110",
    "sliceId": "s1-claim"
  }
}
```

This rewrites the slice status to `done` in the proposal, removes
the lock, and re-syncs the index. Then call `auto_work` again —
it will return the next slice (or `state: "idle"` if the store
is drained).

## Common pitfalls

- **Editing files outside the claim**: `sync_proposals` will
  refuse to mark the slice done. Use a second slice with its own
  claim, or split the proposal.
- **Skipping `sync_proposals`**: the index gets stale. The next
  agent asks for "the next slice" and gets the wrong one.
- **Forgetting to release**: a stale lock blocks the next
  orchestrator for up to `staleMs` (default 30 s). Call
  `proposals_agent_lock { action: "gc" }` to reap.
