# เริ่มต้นใช้งานปลั๊กอิน proposals

บทเรียนนี้เริ่มต้นจาก workspace ที่ว่างเปล่า และจบลงด้วยวงจรที่ทำงานได้
ข้อเสนอ → สไลซ์ → การนำไปใช้ → ปิด โดยยังคงระเบียบวินัย mutex ของไฟล์
สมมติว่าปลั๊กอิน `proposals` เปิดใช้งานแล้ว (ดูที่
`plugins/proposals/README.md` สำหรับ JSON snippet)

## 0. โมเดลความคิด

**ข้อเสนอ** คือไฟล์ markdown ที่มีส่วนหัว frontmatter **สไลซ์** คือส่วนที่
มีหมายเลขอยู่ภายใน ปลั๊กอินประสานงานผู้เขียนสองคนต่อสไลซ์: คนหนึ่งเรียกร้อง
สิทธิ์ อีกคนปล่อย `auto_work` คือจุดเข้าระดับสูง "ฉันควรทำอะไรต่อไป?"

```
docs/mcp-vertex/proposals/
├─ index.json          (สร้างใหม่โดย sync_proposals)
├─ p<N>-<ชื่อ>.md      (ข้อเสนอ)
│  ├─ ## Slices
│  │  ├─ s1-claim
│  │  ├─ s2-implement
│  │  └─ s3-close
```

## 1. เริ่มต้นด้วย `auto_work`

`auto_work` ส่งคืนสไลซ์ถัดไปที่สามารถดำเนินการได้ทั่วทั้ง store ของ
ข้อเสนอ พร้อมแผนที่เรียงลำดับและกระชับ แผนต้องดำเนินการตามตัวอักษร
ไม่ใช่การแก้ไขแบบด้นสด

```json
// การเรียกใช้เครื่องมือ MCP
{ "tool": "proposals_auto_work", "args": {} }

// การตอบสนองทั่วไป (ตัดทอน)
{
  "state": "work",
  "proposalId": "l110",
  "sliceId": "s1-claim",
  "steps": [
    "เปิด docs/mcp-vertex/proposals/l110-…md และเลือกสไลซ์อะตอมิกถัดไป",
    "เรียกร้องสิทธิ์ไฟล์: proposals_agent_lock { action: \"claim\", … }",
    "นำไปใช้เฉพาะสไลซ์นั้น — ไม่มีอะไรนอกไฟล์ที่เรียกร้องสิทธิ์",
    "ตรวจสอบตาม gate ของโครงการ (ดู get_validation_matrix หากมี)",
    "ทำเครื่องหมายความคืบหน้าในข้อเสนอ จากนั้น proposals_sync_proposals",
    "ปล่อย: proposals_agent_lock { action: \"release\", task_id }"
  ]
}
```

## 2. เรียกร้องสิทธิ์ไฟล์ของสไลซ์

เครื่องมือ `proposals_agent_lock` บันทึกว่าใครเป็นเจ้าของเส้นทางใดในช่วง
สไลซ์ หากไม่มีการเรียกร้องสิทธิ์ `sync_proposals` จะปฏิเสธการทำเครื่องหมาย
สไลซ์ว่าเสร็จสิ้น

```json
{
  "tool": "proposals_agent_lock",
  "args": {
    "action": "claim",
    "files": [
      "apps/web/src/components/PluginPage.astro",
      "apps/web/src/data/capabilities.json"
    ]
  }
}
```

การตอบสนองพร้อม `task_id` ที่ต้องเก็บไว้จนกว่าจะปล่อย เอเจนต์สองตัว
เรียกร้องสิทธิ์ไฟล์เดียวกัน ⇒ ขัดแย้ง ไม่มีความคืบหน้า mutex ได้รับการ
สนับสนุนโดย fs (ไม่ใช่ advisory) และอยู่รอดจากการรีสตาร์ทของกระบวนการ

## 3. นำสไลซ์ไปใช้ จากนั้นตรวจสอบ

แก้ไขเฉพาะไฟล์ที่เรียกร้องสิทธิ์ เรียกใช้ gate ของโครงการ:

```bash
bun run validate
```

หาก gate ล้มเหลว ให้แก้ไขสไลซ์ — อย่าขยายการเรียกร้องสิทธิ์โดยไม่เงียบ

## 4. ทำเครื่องหมายความคืบหน้าและซิงโครไนซ์

`sync_proposals` อ่านไฟล์ข้อเสนอ ตรวจสอบ frontmatter + แผนสไลซ์ และ
สร้าง `index.json` ใหม่ เป็นการดำเนินการที่ถูกและ idempotent

```json
{ "tool": "proposals_sync_proposals", "args": {} }
```

## 5. ปิดสไลซ์

```json
{
  "tool": "proposals_close_slice",
  "args": {
    "proposalId": "l110",
    "sliceId": "s1-claim"
  }
}
```

สิ่งนี้เขียนสถานะสไลซ์ใน proposal ใหม่เป็น `done` ลบ lock และซิงโครไนซ์
index ใหม่ จากนั้นเรียก `auto_work` อีกครั้ง — จะส่งคืนสไลซ์ถัดไป (หรือ
`state: "idle"` หาก store หมดแล้ว)

## ข้อผิดพลาดที่พบบ่อย

- **แก้ไขไฟล์นอกการเรียกร้องสิทธิ์**: `sync_proposals` จะปฏิเสธการทำ
  เครื่องหมายสไลซ์ว่าเสร็จสิ้น ใช้สไลซ์ที่สองพร้อมการเรียกร้องสิทธิ์ของ
  ตนเอง หรือแบ่งข้อเสนอ
- **ข้ามไป `sync_proposals`**: index ล้าสมัย เอเจนต์ถัดไปขอ "สไลซ์ถัดไป"
  และได้รับสไลซ์ที่ผิด
- **ลืมปล่อย**: lock ที่ล้าสมัยจะบล็อกผู้ประสานงานถัดไปนาน `staleMs`
  (ค่าเริ่มต้น 30 วินาที) เรียก `proposals_agent_lock { action: "gc" }`
  เพื่อล้างข้อมูล
